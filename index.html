A practical convention that helps researchers automatically document substantive AI use. Once the instructions are installed, a file-capable AI assistant maintains a dated, human-readable history alongside the research project. The convention works across disciplines, project types, and tools.

For Any Researcher and Any Research Project

This guide is for individual researchers, research teams, students, and staff in any discipline. It can be used for manuscripts, data analysis, code, literature reviews, presentations, grant proposals, research administration, teaching materials, and other research-related work.

The goal is not to prescribe a product or require researchers to save entire private conversations. It is to keep a concise, durable activity record close to the project so authors, collaborators, reviewers, or future researchers can understand how AI contributed.

What the AI Records Automatically

After the one-time setup, the instructions tell the AI assistant to create or update the project history at the end of each substantive task. A useful record includes:

  • what the researcher asked the AI to do;
  • which files, data, or external sources the AI used;
  • what the AI inspected, decided, changed, or produced;
  • which checks were performed and what they found; and
  • any assumptions, limitations, failures, or unresolved questions.

Detailed model information—without extra work for users

The AI assistant should also record the exact model and runtime settings automatically. Users should not need to name the model or reasoning setting in every prompt. The assistant should use available task or runtime metadata and the model visibly selected in its interface.

Example for Codex

Model display name: GPT-5.6 Sol
Model family: GPT-5.6
Model slug: gpt-5.6-sol
Reasoning level/effort: Light
Reasoning raw/config value: low
Speed mode: Not exposed in this session
Service tier: Priority

The record should preserve every known field precisely. It should use Not exposed in this session only for an individual field that is genuinely unavailable, never as a substitute for a model or setting that the runtime, interface, or existing conversation already identifies. The same rule applies to Codex, Claude, and other file-capable AI assistants.

For projects with multiple repositories, the same entry should identify the common project root; manuscript and analysis repository branches, commits, remotes, and dirty states; external sources and access dates; journal requirements; files staged, committed, or pushed; and concrete verification evidence such as logs, checksums, archive tests, page counts, or file sizes when relevant.

The history should also record substantive inspection or advice when no project file changes. It supplements—not replaces—discipline-specific disclosure, data-protection, authorship, ethics, funder, institutional, journal, and repository requirements.

One Research Project, One Active History

This guide follows the architecture in the Replication Package Guide. When a project has separate manuscript and analysis repositories, open their common parent as the single Codex or Claude workspace and keep exactly one active project_history/ there.

Research-Project/
├── AGENTS.md
├── README.md
├── manuscript/                 # Git repository 1
├── analysis/                   # Git repository 2
├── project_history/
│   └── Your_Name/
│       └── YYYY-MM-DD by Agent.md
└── others/                     # optional

Do not create competing histories inside the manuscript repository, analysis repository, Overleaf mirror, public replication package, or others/. If an Overleaf or single repository folder is the entire project, that established folder remains the project root.

Worked Example: Common Project Root + Two Repositories

The author’s current workflow keeps the manuscript and analysis in separate Git repositories under one approved, cloud-synchronized research-project root. The manuscript repository may connect to Overleaf. The private analysis repository holds code, validation, generated results, and public-package staging. The common root holds the project map, shared instructions, and private AI-use history.

  1. Create or identify the common research-project root. Keep a root AGENTS.md, a private project-map README.md, the manuscript and analysis repositories, one project_history/, and optional non-core materials.
  2. Install the shared Codex instructions. Copy the setup prompt and paste it into Codex. Codex should save the global instruction file automatically.

    The setup preserves any existing AGENTS.md instructions and merges the recording convention into them; it does not replace the user's file.

  3. Open the common parent folder as the single Codex or Claude project so the agent can inspect both repositories and maintain one history.
  4. Prompt normally. After substantive work, the agent should append to the common dated history automatically, including tasks that only inspect, diagnose, decide, or verify.

Keep Four Artifacts Distinct

  1. Private research/build workspace: the complete authorized environment for analysis and validation.
  2. Manuscript and Overleaf repository: compilation-relevant paper and supplement materials.
  3. Public replication package: a deliberately staged, de-identified, self-contained release.
  4. Journal production and submission files: submission snapshots, proofs, forms, and production exports.

The project history is private by default and should not be copied into the public replication package, manuscript repository, or journal submission unless the project explicitly requires it. Raw or restricted data should remain outside Git and outside an Overleaf-synchronized folder.

Replication-Package Tasks

Before substantive replication-package work, the user and agent should read the current Replication Package Guide, the root AGENTS.md and README.md, and the target journal’s most up-to-date official replication-package instructions. Record each source and access date in the project history.

Journal instructions take priority for package requirements. This AI Collaboration Guide and the root project instructions govern AI-use recording and the location of the single project history.

Adapt It to Other Tools

The recording standard is tool-agnostic. Researchers can adapt the setup details to the AI assistant, storage service, document platform, or repository they already use.

  • Codex or Claude Code: use either included setup prompt. Both install the same recording convention in the location read by that tool.
  • Direct or GitHub-connected Overleaf: keep one common history above the manuscript folder when the manuscript is part of a larger project; otherwise use the established manuscript-project root.
  • Dropbox, Google Drive, or iCloud: use only institutionally approved storage and respect data agreements. Cloud sync and Git have different scopes.
  • Another research workflow: keep one history at the established root in a shared, durable, approved location.
  • Tools without file access: export or copy a concise activity record into the project history at the end of substantive work.

Automatic recording requires an AI tool that can write files and read persistent instructions. With other tools, the same record may need to be exported or saved manually.

Project History Convention

Unless a project has its own convention, the AI tool should create or append to this file at the established common project root:

project_history/Your_Name/YYYY-MM-DD by Codex.md
project_history/Your_Name/YYYY-MM-DD by Claude.md

Use one folder per human researcher and one file per date and AI tool. Append each substantive task to that day's file instead of overwriting earlier entries. Every appended entry should retain detailed, task-specific model metadata because the model or reasoning setting may change during the day. For solo projects, the researcher is also the history owner. Do not place confidential, sensitive, restricted, proprietary, or identifying information in the history unless the project permits it.
